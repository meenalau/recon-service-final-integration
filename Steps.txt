Problem

Performance tuning means:

2. Memory Management + Garbage Collection
Problem

Every request creates objects.

Example:

Trade trade = new Trade();

Thousands of requests create thousands of objects.

JVM Heap fills up.

Solution

Garbage Collector (GC)

Automatically removes unused objects.

Benefits:

Prevent memory leak
Avoid OutOfMemoryError

You don't need coding here.

Mostly interview/theory topic.

3. Spring Boot Actuator
Problem

How do we know application health?

Without Actuator:

No visibility.

Solution

Add Actuator

Dependency:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

Useful URLs:

/actuator/health
/actuator/info
/actuator/metrics
/actuator/env

Examples:

http://localhost:8080/actuator/health

Response:

{
  "status": "UP"
}
Why Needed?

Production monitoring.

4. DB Query Optimization
Problem

Database is usually slowest component.

Example:

SELECT * FROM trades

returns 1 million rows.

Bad.

Indexes

Without index:

SELECT *
FROM trades
WHERE trade_id=100;

DB scans whole table.

With index:

CREATE INDEX idx_trade_id
ON trades(trade_id);

Search becomes much faster.

N+1 Problem

Common JPA issue.

Example:

Load 100 trades.

Then for each trade load instrument.

1 query for trades

+
100 queries for instruments

Total:

101 queries

Called N+1 problem.

Solution:

JOIN FETCH

or

@EntityGraph

Mostly theoretical for Sprint.

5. Caching Strategies Overview
Problem

Suppose instrument rarely changes.

API:

GET /instruments/IBM123

1000 users call it.

Without cache:

1000 DB hits

Bad.

With Cache

First call:

DB Hit

Second call:

Cache Hit

No DB call.

Faster.

MODULE 6 : Monitoring, Logging & Caching
1. Caching using @Cacheable

This is your Sprint 1 task.

Current
InstrumentService.findByIsin()

Every call hits DB.

Required
@Cacheable("instruments")
public Instrument findByIsin(String isin)
{
    return repository.findByIsin(isin);
}

Meaning:

First call:

DB

Second call:

CACHE
